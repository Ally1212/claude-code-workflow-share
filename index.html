掌握副驾驶。

这不是一篇“插件越多越强”的清单，而是一套把 Claude Code 调成真正生产力系统的思路：先降低摩擦，再建立规则，再持续沉淀， 最后让你的 AI 副驾驶越来越像你。

看核心框架 看我的技能栈
“

理论上你可以学会任何东西。前提不是模型够强，而是你有没有把权限、工作流、技能、记忆和经验库真正串起来。

从“会用”到“会调”，再到“会沉淀”。
01 / Framework

先把框架搭对，再谈效率。

这套方法的底层很简单，只有三层：第一层是权限，减少来回确认；第二层是规则，让 Claude Code 记住你的偏好；第三层是沉淀，把每次踩坑、流程、结论都留下来，形成复利。

Layer 01 权限

自定义一个像 `yolo` 这样的启动命令，本质是先把交互摩擦降到最低，让思路不断线。

Layer 02 规则

用全局配置约束语言、调研偏好、回答风格和编码原则，控制模型默认行为，而不是每次重说。

Layer 03 沉淀

把错误、经验、部署流程、学习心得、项目记忆都保存下来，让下一次协作从更高起点开始。

02 / Setup

好的开始，是把 Claude Code 调成你的样子。

全局配置不需要复杂，关键是只保留跨项目、长期稳定、你真的会反复用到的规则。越短越好，越准越好，最好控制在一屏能看完的规模。

Command

先做一个顺手的启动入口

例如 `yolo` 这类命令，目的是一键拉起 Claude Code 并打开你常用的权限范围。不是为了炫技，而是为了让“想做”到“开始做”之间没有阻力。

CLAUDE.md

把模型长期记住的东西，写进全局配置

  • 始终中文回答，但联网调研优先英文高质量信息源。
  • 只写跨项目规范，项目特定约束放项目自己的 `CLAUDE.md`。
  • 保持精简，不让全局记忆被噪音淹没。
03 / Workflow

真正的提升，不来自一次回答，而来自一套循环。

每次开始编码前，先问有没有现成技能；每次出错后，记下这次教训；每次流程跑通后，把它沉淀成文档或记忆。这样 Claude Code 才会从“回答问题”逐步变成“长期协作者”。

Before Code

先找技能，不要重复造轮子

要接入支付，就先用 `/find-skills` 找 RevenueCat 相关技能；要做部署、设计、测试、性能优化，也先查有没有现成工作流。先装配，再开工。

After Fix

问题解决后，把经验写回项目记忆

出错并不可怕，可怕的是下次继续犯。把这次错误原因、修复方式、避免建议记录进项目记忆，让模型下一轮更少走弯路。

After Repeat

把高频流程沉淀成 `docs`

部署、上线、回滚、环境配置、服务端脚本，只要会重复发生，就写成项目内文档。下一次直接让模型参考文档执行，而不是从头口述。

Knowledge

外部笔记，是你的外挂知识库

像 `/biji`、`/learn` 这样的命令，本质上是在把会话里的洞见、开源项目拆解、方法论总结，持续写进 Obsidian 这样的本地知识库。

04 / Stack

技能不是越多越好，而是越贴合你的工作越好。

我更推荐先装两个“元技能”：一个负责找技能，一个负责造技能。剩下的插件和技能，都围绕你自己的项目结构、技术栈和高频问题来配置。

05 / Environment

环境也要服务于思考。

一个透明终端、多个分屏、顺手的快捷键、关键节点的 hooks、必要时的 worktree，这些并不是装饰，它们共同决定了你和模型协作时的节奏感。

Terminal

让终端像工作台，而不是黑盒

例如使用 Kaku 这类可定制、支持多窗口与透明效果的终端，把多个上下文同时展开，会比频繁切换窗口流畅很多。

Hooks & Worktrees

复杂任务再上自动化拆分

hooks 适合在关键节点自动触发检查与提示；多 worktree 适合大任务拆分，但是否保留，要看它是不是让你更轻松，而不是更复杂。

06 / Closing

最终目标，不是更会聊天，而是更会造产品。

模型会越来越强，但真正拉开差距的，是你有没有持续打磨自己的系统。权限、技能、规则、记忆、流程、文档、复盘，这些东西一旦连起来，Claude Code 才会越来越像一个“懂你”的副驾驶。

使用最强模型，做心里真正想做的产品。

这套方法没有神秘感。它只是要求你认真对待自己的工作流，然后不断微调。你不是在适应工具，而是在训练一个越来越懂你的开发伙伴。

一句话总结

先降低启动摩擦。

再把常用规则写清楚。

最后把经验系统性沉淀下来。
